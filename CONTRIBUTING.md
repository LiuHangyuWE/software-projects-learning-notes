# 贡献指南

首先，非常感谢您愿意为这个项目花费宝贵的时间！

本项目的创建初衷与大多数开源项目不同。它不是一个社区共建的百科全书，而是我个人的学习路径、思考记录和顿悟集锦。

本仓库的核心价值在于原汁原味和个性化深度。我希望它能真实地反映我个人的学习轨迹和思考过程，以此为寻求路径的同学提供一个真人样本。

因此，本项目的贡献策略**严格**围绕这一核心战略制定。请您在贡献前仔细阅读以下指南。

## 行为准则 (Code of Conduct)

为了营造一个友好、尊重的交流环境（尤其是在 Issue 区），我们希望所有参与者都能遵守 [行为准则](./CODE_OF_CONDUCT.md)。请在参与讨论前阅读它。

## 提交 Issue (讨论问题)

我们非常欢迎您通过 Issue 来讨论。

* **Bug 报告 / 内容纠错：** 如果您发现我现有的笔记中有事实性错误、拼写错误、代码Bug或失效链接，请不要犹豫，立刻提交 Issue 报告它！

* **改进建议 (非内容性)：** 如果您对我现有的笔记内容（例如手搓OS或元知识部分）有关于如何讲得更清楚的建议（例如这里的推理可以更清晰或这个比喻不恰当），也非常欢迎您通过 Issue 提出，我们可以一起讨论。

* **关于新知识点的 Issue：** 本项目严格服务于我的个人学习路径。如果您有添加新知识点的建议，请理解，我（作者）会对其进行审核。我只会考虑添加那些符合我当前学习规划和兴趣的内容。任何与我路径无关的新知识添加请求，我（作者）保留不予采纳的权利。

## 提交 PR (Pull Request) 的工作流

我们主要接受用于修正的 PR。提交 PR 的标准流程如下：

1. **Fork** 本仓库到您的个人账户下。

2. **Clone** 您的 Fork 到本地。

3. 从 `main` 分支创建您的修复分支。请规范命名您的分支，例如：

   * `fix/typo-in-readme` (修复README的拼写错误)

   * `fix/broken-link-in-os-notes` (修复OS笔记中的失效链接)

4. 在本地进行修改并 Commit。

5. Push 您的修复分支到您的 Fork 仓库。

6. 在 GitHub 上，从您的修复分支向本仓库的 `main` 分支发起一个 Pull Request。

7. 请在 PR 描述中清晰说明您修复了什么问题，并关联对应的 Issue（如果有的话）。

## 编码规范

本项目包含的 .md 文档和少量代码（如手搓OS部分）。

* **Markdown 文档：** 请保持排版简洁清晰，遵循基本的 Markdown 语法。

* **代码：** 提交的代码（包括PR中的代码修复）应尽量遵循 [《阿里巴巴Java开发手册》](https://github.com/alibaba/Alibaba-Java-Coding-Guidelines) 的规范。虽然本项目不全是Java，但我们以此为标准，追求高质量和规范的代码。

## 贡献类型

为确保本项目的战略定位，我们对 PR 类型有严格的区分：

#### ✅ 欢迎并接受的 PR 类型：

* 拼写错误、错别字修正。

* 对现有代码示例中的 Bug 进行修复（请遵循编码规范）。

* 失效链接的更新。

* 对现有内容排版、格式的优化（例如修复 Markdown 渲染问题）。

#### ❌ 不接受的 PR 类型（或将被严格审核）

* **任何添加您个人学习笔记或项目总结的 PR：** 这类 PR 将不被接受，因为它违背了项目作者个人路径的核心。

* **任何添加新知识点、新章节或新笔记的 PR：** 这类贡献的门槛非常高。它们必须由我（作者）进行严格审核，以确保它们100%符合我的学习路径。任何我（作者）尚未学习、或不打算学习、或认为不符合我路径的内容，将不被合并。

* **任何大规模重构或删除我（作者）现有核心内容或思考路径的 PR。**

任何不符合上述欢迎类型的 PR 将会被关闭，或在审核后不予合并。这并非不尊重您的劳动，而是为了坚定地执行本项目保持个人原汁原味的核心战略。

感谢您的理解与支持！
